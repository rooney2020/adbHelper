第一章

项目概览：目标 · 定位 · 关键指标

ADB Helper 是一个跨平台桌面应用（Ubuntu / Windows / macOS），核心目标是将全部 adb 命令能力组织为可视化工作流，让用户无需记忆复杂命令行参数即可完成所有 adb 操作，并对不同 ROM 和 Android 版本的兼容性做显式识别与降级处理。

核心设计目标

目标 01 · 全量覆盖

全部 adb 能力可视化

不靠静态按钮堆叠，通过"分类目录 + 搜索 + 收藏 + 原始命令模式"组合承载全部 adb 命令族，覆盖 adb、shell、am、pm、ime、content、wm、dumpsys、input、svc 等全部子命令。

目标 02 · 兼容适配

Capability 优先执行

每条命令执行前自动探测设备 Capability，识别命令可用性、参数兼容性和降级替代方案。针对不同 Android 版本和厂商定制 ROM 给出显式的"支持 / 受限 / 不支持"状态。

目标 03 · 安全隔离

前后端三层架构

Renderer 层禁止直接访问 Node 能力或系统命令，所有执行请求经 Electron 主进程白名单 IPC 和 Python 后端适配层。高风险写操作触发二次确认弹窗。

核心理念

Capability 优先：所有 adb 操作必须先经过设备识别、能力探测和适配决策，禁止 UI 直接盲执行底层命令。当命令在特定 ROM / Android 版本上不可用时，系统自动给出降级替代方案和原因说明。

架构原则

前后端隔离：Renderer 禁止直接访问 Node 能力或系统命令，所有执行请求必须经 Electron 主进程白名单 IPC 和 Python 后端适配层。
Capability 优先：所有 adb 操作必须先经过设备识别、能力探测和适配决策，禁止 UI 直接盲执行底层命令。
结果双视图：文本型命令必须同时保留原始输出与结构化视图，解析失败时禁止丢失原文。
全量可导航："全部 adb 命令"必须通过分类目录、搜索、收藏和原始命令模式组合承载，禁止全部能力直接平铺成按钮堆叠。

结果分析能力

能力 01

关键词搜索 + 高亮

全文本关键词搜索，匹配项高亮显示，支持正则表达式。在大体量 dumpsys / logcat 输出中快速定位关键信息。

选择任意两条历史记录进行逐行 diff，支持 token 级差异高亮。对比不同设备或不同时点的命令输出差异。

结果双视图硬规则：文本型命令必须同时保留原始输出与结构化视图，解析失败时禁止丢失原文。用户可在两种视图之间一键切换，确保任何时候都不会因解析失败而无法查看结果。

架构决策记录

ADR-001

为何选择 Electron + React + Python

Electron 满足跨平台桌面分发和复杂文本展示需求；React 适合大型信息架构和组件化复用；Python 便于封装 adb 调用、适配规则和工具链。未选 Qt（原生成本高）、未选 Tauri（生态不如 Electron 成熟）。

设计原则

Capability 优先执行 + 全量可导航

所有 adb 操作先探测后执行；"全部 adb 命令"通过分类目录 + 搜索 + 收藏 + 原始命令模式组合承载，避免全量功能一次性显式铺开导致信息架构失控。

风险缓解策略

风险概率影响缓解策略
全量 adb 命令范围过大导致信息架构失控高高分类目录 + 搜索 + 收藏 + 原始命令兜底
不同 ROM/Android 版本命令差异大高高Capability 探测、适配元数据和替代链路
dumpsys 等结果过大造成渲染卡顿中高虚拟滚动、延迟解析、结果分段、原文兜底
Electron 与 Python 进程通信复杂中中白名单 IPC + 明确 CLI 契约 + 契约测试
三平台打包差异影响交付中高提前规划 Electron Builder 配置、平台能力矩阵
