# ADB Helper 需求规格

## 概述

ADB Helper 是一个面向 Android 开发人员与测试人员的跨平台桌面应用，运行于 Ubuntu、Windows、macOS，用于把全部 adb 能力组织为可视化工作流，并对不同 Android 版本、厂商定制 ROM 与设备能力差异做显式识别、降级处理和结果说明，降低命令记忆成本与误操作风险。

## 功能需求

### 核心功能

- **FR-001**: 系统必须提供设备连接管理视图，展示已连接设备、序列号、连接状态、Android 版本、品牌/机型、当前传输方式（USB/Wi-Fi）及基础 capability 摘要。
- **FR-002**: 系统必须在设备选中后执行 adb capability 探测，识别命令是否可用、参数是否兼容、替代命令是否存在，并以“支持 / 受限 / 不支持”的形式展示。
- **FR-003**: 系统必须为常用查看型命令提供结构化结果视图，至少覆盖 dumpsys、logcat 抓取、am / cmd 查询类命令以及系统属性读取，并支持原始文本与解析视图切换。
- **FR-004**: 当用户执行某个 adb 操作时，系统必须先校验目标设备 capability，并在命令不可用时给出原因、替代命令或降级建议，而不是直接执行失败。
- **FR-005**: 系统必须支持覆盖全部 adb 命令族的命令目录与模板库，允许用户按“设备、shell、am、pm、cmd、logcat、dumpsys、input、settings、文件、网络、fastboot 扩展”等分类快速发起操作。
- **FR-006**: 系统必须支持结果过滤、关键字高亮、分段折叠、字段对齐和导出，帮助用户阅读大体量 dumpsys / stack / activity 输出。
- **FR-007**: 系统必须记录最近执行历史，包括设备、命令模板、执行时间、结果状态和摘要，允许一键复用最近操作。
- **FR-008**: 系统必须在执行高风险写操作或状态变更操作前进行二次确认，并明确展示影响范围。
- **FR-009**: 系统必须提供“原始命令模式”，允许用户执行尚未模板化的 adb 命令，并在执行前仍然完成设备选择、能力预检、参数提示和风险提示。
- **FR-010**: 系统必须为全部纳入目录的 adb 命令维护适配元数据，至少包含适用 Android 版本、常见 ROM 差异、权限前提、常见失败原因和可选替代命令。

### 扩展功能

- **FR-101**: 系统应支持用户维护自定义命令模板，并为模板声明适用条件、前置检查和结果展示方式。
- **FR-102**: 系统应支持多设备并行查看模式，便于测试人员对比不同 ROM 或不同 Android 版本的命令返回。
- **FR-103**: 系统应支持将关键结果导出为 Markdown、JSON 或纯文本，便于缺陷单、测试报告和研发分析复用。
- **FR-104**: 系统应支持离线查看最近一次成功抓取的结果快照，用于断开设备后的继续分析。
- **FR-105**: 系统应支持对命令目录进行搜索、收藏和最近使用排序，降低“全部 adb 命令”规模带来的查找成本。

## 非功能需求

| 维度 | 要求 | 量化指标 |
|------|------|---------|
| 性能 | 常规命令触发和结果展示需保持流畅 | 单设备普通查询首次响应 < 1.5s；10 万字符结果视图切换 < 500ms；命令目录搜索响应 < 200ms |
| 兼容性 | 支持主流桌面系统和多 Android 版本设备 | Ubuntu 22.04+、Windows 10/11、macOS 13+；Android 8-15 设备可接入 |
| 可靠性 | adb 执行失败需可恢复、可追踪 | 命令失败后 100% 返回错误摘要与重试入口；最近 50 次历史可保留 |
| 安全 | 高风险命令要隔离和确认 | 默认禁止无确认执行写操作；无硬编码敏感信息 |
| 可维护性 | 命令适配、解析器、界面分层清晰 | capability 适配层、命令执行层、结果解析层、界面层职责分离 |
| 可观测性 | 关键操作具备日志与诊断信息 | 每次执行记录设备、模板、返回码、耗时、stderr 摘要 |

## 用户场景

### 主流程

1. 用户连接一台 Android 设备并启动应用。
2. 系统识别设备并执行 capability 探测，生成支持矩阵。
3. 用户从命令模板库选择“查看任务栈”或“抓取 dumpsys activity”。
4. 系统根据设备 capability 选择可用命令或降级替代方案。
5. 系统展示结构化结果，并允许搜索、高亮、折叠和导出。

### 替代流程

1. 当目标命令在某 ROM 上不存在时，系统自动切换到兼容命令或提示手动候选方案。
2. 当用户不熟悉 adb 原生命令时，可直接从分类模板中选择目标操作，无需手动输入。

### 异常流程

1. 当设备离线、无授权或 adb server 异常时，系统必须提示原因、恢复建议和重试入口。
2. 当命令输出无法解析为结构化数据时，系统必须保留原始文本视图，并标注解析失败原因。
3. 当用户触发高风险命令时，系统必须先确认设备和影响，再允许继续执行。

### 恢复流程

1. 当命令执行中断或设备断开后，系统应允许用户重新连接并恢复最近一次结果快照查看。

### 边界情况

- 多设备同时连接但 capability 不一致，且同名命令支持度不同。
- 厂商定制 ROM 删除或改写标准 adb / shell 子命令。
- 结果文本超长、字段嵌套深、输出结构不稳定。
- 用户机器本地 adb 版本与设备侧能力不匹配。
- 全量命令目录过大，用户需要通过搜索、收藏与分类迅速定位目标命令。

## 假设与约束

- **假设**：用户已在本机安装可用 adb，且有权限连接目标 Android 设备。
- **假设**：首版聚焦单人桌面工具，不涉及多人协作或云端共享。
- **约束**：不同设备 shell 环境、ROM 定制与权限模型差异大，命令适配必须以内建 capability 探测为前提。
- **约束**：由于你要求全部 adb 命令均可通过工具访问，首版必须采用“分类目录 + 搜索 + 原始命令模式 + 适配元数据”的组合方案，避免仅靠静态按钮堆叠。
- **约束**：项目目录固定为当前目录，文档、代码、测试与打包产物均需在本工程内组织。

## 主动补充需求

1. **结果快照离线回看**：针对测试定位场景，增加最近一次抓取结果离线回看能力。
2. **命令历史复用**：针对高频 adb 操作，增加最近执行历史和一键重放。
3. **高风险操作确认**：对非查看型命令增加显式风险提示和二次确认。
4. **自定义模板声明**：允许团队把内部常用 adb 组合沉淀为模板并标注适用条件。
